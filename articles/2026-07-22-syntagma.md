Syntagma is an agentic curriculum lifecycle management system built for PES University, Bengaluru. It replaces a manual, error-prone process — faculty submitting raw course content in inconsistent formats, someone manually compiling it into the official syllabus document, no version history, no way to compare changes between years.

The system spans the full pipeline: faculty submit course content through a form, AI refines it into structured records, an agentic editor with 36 tools lets admins propose and review changes, the curriculum renders as official A4 PDFs with letterhead, and every version is snapshotable with full diff history.

## Architecture Overview

The system has three layers:

1. **Backend** — FastAPI 0.138 on Python 3.12, 51 endpoints across 9 route files
2. **Frontend** — Vanilla HTML/CSS/JS, zero build step, served as static files
3. **Infrastructure** — Supabase (PostgreSQL) for persistence, Upstash Redis for cache, OpenRouter for AI, Docker on Hugging Face Spaces, Vercel for frontend

```
Browser → Vercel (static) → /api/* → HF Spaces (FastAPI Docker)
                                       ├── OpenRouter (LLM)
                                       ├── Upstash Redis (cache)
                                       ├── Supabase Postgres
                                       └── WeasyPrint (PDF)
```

Frontend surfaces: `/auth/` (sign in), `/` (dashboard), `/form/` (course submission), `/courses/` (management table), `/preview/` (PDF viewer), `/live-editor/` (agentic editor), `/versions/` (version history), `/docs/` (documentation).

## The Submission Pipeline

Faculty enter a course code, title, and raw content through a form. The backend auto-parses course codes — semester, department, and credit category are extracted from the code format itself (e.g., `UE25CS242B` → semester 4, CS department, credit category 2).

A background task triggers AI refinement via OpenRouter. The LLM receives the raw faculty text and returns structured JSON matching the database schema — preserving all syllabus topics, only cleaning and structuring. The raw content is never lost; it lives alongside the refined record.

## The Agentic Editor

The most architecturally interesting part. The editor is an AI chat that has access to 36 tools for reading and writing curriculum data, streamed via Server-Sent Events.

**The critical design rule: the agent never auto-applies changes.** Every proposed edit creates a `proposed` draft in the `agent_drafts` table. The human must review and approve in the Review panel before anything touches `refined_submissions`. This is enforced at every level:

- `create_course_draft` creates a reviewable draft for one course
- `create_document_draft` creates a multi-course draft
- `update_deterministic_fields` creates a `blocked` draft for protected fields (program, hours, credits, course_type)
- Drafts have statuses: `proposed` → `applied`, or `blocked` if they touch protected fields
- Applying a draft writes to `course_revision_history` first (audit trail), then updates `refined_submissions`, then auto-creates a version snapshot

The SSE stream is also interesting — it sends `token` events for text, `draft` events when a draft is created (so the frontend can update the Review panel in real time), `refined_course` events for new courses, and `error` events with graceful fallback.

## Version Snapshots

The versioning system is effectively git for curriculum data. `curriculum_versions` stores named snapshots (conventional-commit-style names generated from diff summaries). `finalized_submissions` stores a JSON snapshot of every course at the time of version creation.

Features:
- Version creation detects no-change and rejects duplicate snapshots (409 Conflict)
- Version restore creates full revision history and archives courses removed since the snapshot
- Version-vs-version diff shows added, removed, and changed courses with per-course change percentages
- Cache invalidation on any version operation

## The PDF Pipeline

Curriculum PDFs render through Jinja2 templates → WeasyPrint → A4 PDF with PES University's letterhead (base64-encoded in `_logo_data.py`). There are four template variants:

- `jinja_sample.html` — Single course, semester, or full curriculum layout with summary tables
- `jinja_diff.html` — Red/green side-by-side diff view
- `jinja_program.html` — Program educational objectives and outcomes
- `jinja_sem_5_6.html` — Specialization track brackets for semesters 5-6

A background thread pre-generates PDFs at startup for configured academic years, so the first request doesn't pay the cold-start cost.

## Agent Tools (36)

The agent tools are defined as a registry of dataclass instances — each with a name, description, JSON Schema parameters, and a handler function. The tool listing is exposed at `GET /agent/tools` and ingested by OpenRouter's function-calling API.

Categories:
- **Read** (10 tools): granular course field access (`get_course_syllabus`, `get_course_textbooks`, `get_course_lab`, `get_course_fields`, `batch_read_courses`)
- **Write** (6 tools): `create_course_draft`, `update_agent_draft`, `create_refined_course` (for new courses or draft-status updates), `create_document_draft`, `update_deterministic_fields`, `create_curriculum_version`
- **Diff** (3 tools): `diff_course_json`, `diff_versions`, `get_version`
- **Specialization** (7 tools): track definitions, elective assignment, AI-driven categorization
- **External** (3 tools): `fetch_url`, `web_search` (DuckDuckGo), `get_attachment_text`
- **Output** (2 tools): `create_report` (markdown/PDF), `create_spreadsheet` (CSV/XLSX)
- **Meta** (4 tools): `signal_done`, `ask_user`, `list_courses`, `get_preview_url`

## Caching

The cache layer (`cache.py`, 134 lines) wraps both Upstash Redis and an in-memory dict behind the same interface. If Redis is unavailable, it falls back silently to in-memory. Lazy invalidation clears specific cache keys on writes — course visibility changes invalidate `courses_list`, `sem_courses:*`, `pending_courses`, `all_course_ids`, and the full curriculum cache.

A cache prewarm thread runs at startup to populate `all_course_ids`, avoiding cold-start latency on the first request.

## OpenRouter Integration

The LLM client (`openrouter.py`, 475 lines) handles:
- **Tool calling** — Streams token-by-token, intercepts tool_calls, executes them, sends results back
- **Fallback model** — If the primary model returns 502/503 with certain provider messages, it switches to the fallback automatically
- **Fibonacci backoff** — On transient errors, retries with delay doubling (1s, 2s, 3s, 5s, 8s...)
- **Context length management** — Fetches model's actual context_length at startup, trims oldest messages when budget is exceeded
- The system is model-agnostic; it works with any OpenAI-compatible endpoint

## Database Schema

12 tables in Supabase PostgreSQL:

- `submissions` — Raw faculty input with status lifecycle: `pending` → `refined` / `refine_failed`
- `refined_submissions` — Template-ready courses with 22 columns including JSONB `units`
- `specialization_definitions` — Tracks per semester with letter/key/name
- `course_specialization_assignments` — Many-to-many with unique constraint
- `curriculum_versions` — Snapshots with conventional-commit-style names
- `finalized_submissions` — Course JSON snapshots per version
- `course_revision_history` — Full audit trail of every applied change
- `agent_drafts` — Proposed/blocked/applied lifecycle
- `agent_document_drafts` — Multi-course drafts
- `chat_sessions`, `chat_messages` — Chat with metadata (tool calls, attachments)
- `chat_attachments` — Uploaded files and generated reports with text extraction

## Design Decisions

**Vanilla frontend, no framework.** The frontend is intentionally plain HTML/CSS/JS. No React, no Vue, no build step. The backend serves it as static files. This was a deliberate choice — for a tool used by a small number of curriculum admins, the overhead of a modern SPA framework is unjustified. Every page is a standalone HTML file with inline logic.

**No async task queue.** Celery or similar was considered and rejected. The requirements stated that only a few people would use the system simultaneously. Background refinement runs in FastAPI's `BackgroundTasks`, PDF pre-generation runs in a daemon thread at startup. Both are synchronous and fast enough for the expected load.

**Model-agnostic AI pipeline.** The refinement and agent systems work with any OpenAI-compatible API. The model is configurable via environment variables (`OPENROUTER_MODEL`, `OPENROUTER_FALLBACK_MODEL`). This prevents vendor lock-in and allows swapping models as the ecosystem evolves.

**Drafts as first-class entities.** The agent never directly modifies a course. Every change goes through `agent_drafts`, gets a `diff_summary` (JSON Patch + syllabus change percent + topic add/remove lists), and waits for human approval. This was the most important architectural constraint — it makes the AI a proposal tool, not an autonomous actor.
