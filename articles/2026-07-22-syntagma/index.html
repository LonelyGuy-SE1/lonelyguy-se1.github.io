Syntagma, Building An Agentic Curriculum Lifecycle Management System

back to blogs

A production-grade curriculum management platform for PES University: agentic refinement, 36-tool editor with reviewable drafts, git-like versioning, Jinja2-powered A4 PDFs, and a deliberately vanilla frontend.

Syntagma replaces a completely manual process at PES University. Faculty submit course content in raw, inconsistent formats. Someone manually compiles it into the official syllabus document. No version history, no way to compare changes between years. The entire pipeline is now automated end-to-end.

Frontend Surfaces

The application has 8 frontend routes, all vanilla HTML/CSS/JS served as static files from Vercel. API calls are proxied through Vercel rewrites to the Docker backend on Hugging Face Spaces with CORS middleware handling the bridge.

Dashboard at the root route shows a navigation grid and academic year selector. Each route is a standalone HTML file with its own CSS and JS, no framework overhead.

Course Submission

Faculty enter a course code, title, and raw course content. The backend auto-parses course codes: UE25CS242B is decomposed into semester 4, CS department, credit category 2, target department CSE. Background tasks trigger agent refinement without blocking the response.

Course Management

The courses table supports semester and visibility filters, visibility toggles (show/hide from curriculum without deletion), and soft-delete via archival. All mutations invalidate the relevant cache keys to maintain consistency.

The Agentic Editor

This is the core architectural piece. The editor surfaces three panels:

  1. Chat panel -- SSE-streaming conversation with the agent. The agent receives a system prompt that encodes curriculum rules (credit categories, course code conventions, unit hour requirements, specialization track structure) plus the active course context.
  2. Fields panel -- Direct JSON field editing for quick manual adjustments.
  3. Review panel -- Lists proposed drafts with diff summaries, change reasons, and apply/block actions.

The critical rule: the agent never auto-applies changes. Every proposed edit creates a row in agent_drafts with status proposed. The human must review and explicitly apply it. Fields tagged as deterministic (program, hours, credits, course_type) create blocked drafts that require extra confirmation. This is enforced at every level of the tool chain.

The Tool System

36 tools registered as a frozen dataclass registry. Each tool has a name, description, JSON Schema parameters (for LLM function calling), and a handler function. The tool listing is exposed at GET /api/agent/tools.

Read tools (10): granular course access from lightweight identifiers (get_course_codes) to full JSON (get_current_course_json), with targeted field subsets (get_course_syllabus, get_course_textbooks, get_course_lab, get_course_deterministic, get_course_fields) and batch operations (batch_read_courses).

Write tools (6): create_course_draft (single course, reviewable), create_document_draft (multi-course), create_refined_course (new courses or draft-status updates), update_agent_draft (modify existing draft), update_deterministic_fields (blocked draft for protected fields), create_curriculum_version (named snapshot).

Diff tools (3): diff_course_json (JSON Patch + syllabus change percent), diff_versions (added/removed/changed courses between snapshots), get_version (load snapshot).

Specialization tools (7): define_specialization, list_specializations, assign_elective_to_tracks, remove_elective_from_tracks, get_course_assignments, categorize_elective (AI-driven track assignment).

External tools (3): fetch_url, web_search (DuckDuckGo), get_attachment_text (PDF/DOCX/XLSX/TXT text extraction).

Output tools (2): create_report (markdown or PDF saved as chat attachment), create_spreadsheet (CSV or XLSX from structured row data).

Meta tools (4): signal_done (end turn with summary), ask_user (request clarification), list_courses, get_preview_url.

The tool chain supports complex permutations: the agent can batch-read courses, generate a spreadsheet, research web sources, create document-level drafts across multiple courses, and snapshot a version -- all in a single turn. The type system (frozen dataclass, typed JSON Schema) keeps tool definitions inspectable and parseable by any OpenAI-compatible function-calling API.

PDF Pipeline

PDFs are generated through Jinja2 templates rendered by WeasyPrint. There are 4 template variants:

  • jinja_sample.html -- Single course, semester, or full curriculum with summary tables
  • jinja_diff.html -- Red/green side-by-side diff view
  • jinja_program.html -- Program educational objectives and outcomes
  • jinja_sem_5_6.html -- Specialization track brackets

The PES University letterhead is base64-encoded directly in _logo_data.py and embedded into every PDF. Download options cover single course, per-semester, and full curriculum, each with inline preview and attachment download modes.

A background thread pre-generates PDFs at startup to avoid cold-start latency on the first request.

Version Control System

The versioning system is modeled on the same underlying architecture as git -- for curriculum data. curriculum_versions stores named snapshots. finalized_submissions stores a course_json snapshot for every course at version creation time.

Key behaviors:
- Version creation detects no-change and returns 409 Conflict
- Version restore writes to course_revision_history and archives courses removed since the snapshot
- Version-vs-version diff via GET /versions/{v1}/diff/{v2} returns added, removed, and changed courses with per-course change percentages
- Applying a draft auto-creates a version snapshot with a conventional-commit-style name derived from the diff summary (Apply: updated UE25CS242B syllabus (+3 topics, syllabus:25%))
- Cache invalidation fires on every version mutation

Relationships cascade on delete through foreign keys. The schema is self-documenting via GET /api/health/schema which checks all 12 tables exist with the correct columns.

Caching Layer

The cache module (cache.py, 134 lines) wraps Upstash Redis with automatic in-memory fallback behind a unified interface. If Redis is unreachable, it degrades silently to a thread-safe dict with TTL-based expiration.

Lazy invalidation uses a prefix-based pattern: a course visibility change invalidates courses_list, sem_courses:, pending_courses, all_course_ids, and the full curriculum cache. Version operations invalidate versions_list and ver_preview:. A cache prewarm thread runs at FastAPI startup to populate all_course_ids.

Key Design Decisions

Vanilla frontend, zero build step. Every page is a standalone HTML file with its own CSS and JS. No React, no Vue, no webpack. The backend mounts the frontend directory as a static file server. For a tool used by a small number of curriculum administrators, the overhead of a modern SPA framework is unjustified.

No async task queue. Celery was considered and rejected because the requirements specified a small user base. Background refinement runs in FastAPI BackgroundTasks. PDF pre-generation runs in a daemon thread at startup. Both are synchronous and fast enough for the expected concurrency.

Agent-agnostic AI pipeline. The agent system accepts any OpenAI-compatible API endpoint. The model, URL, and fallback model are all environment-configurable. The system prompt encodes curriculum-specific rules that are independent of any particular model.

Drafts as first-class entities. Every change goes through agent_drafts, gets a diff_summary (JSON Patch + syllabus change percent + topic add/remove lists), and waits for human approval. This was the single most important architectural constraint. It makes the agent a proposal tool, not an autonomous actor. The draft lifecycle (proposed -> applied, or blocked if deterministic fields are touched) is enforced at every level of the tool system.
